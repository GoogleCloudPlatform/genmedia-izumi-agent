## Development & Testing

### `start-all.sh`

The unified launcher that boots both the **React UI** and the **FastAPI Backend Services** simultaneously. This is the recommended entry point for testing the full application locally with one command.

**Usage:**

```bash
./scripts/start-all.sh
```

*   **API Layer:** `http://localhost:8000`
*   **UI Layer:** `http://localhost:5173`

Passes any downstream arguments (e.g., `--with-db-emulator`) safely to the Python server daemon! Aborts both cleanly under a standard `Ctrl+C`.

### `start-local-server.sh`

Starts the local development server using `uvicorn` with hot-reloading enabled. It also allows running with the Firestore emulator. This script is primarily for developers actively working on implementing GenMedia agents. It allows them to run and test their agents locally with hot-reloading, making it ideal for the inner development loop.

**Usage:**

*   **Start with real database (Default):**
    ```bash
    ./scripts/start-local-server.sh
    ```
    This will connect to the Firestore instance configured in your `.env.local` files.

*   **Start with Firestore Emulator:**
    ```bash
    ./scripts/start-local-server.sh --with-db-emulator
    ```
    Use this flag to avoid contaminating the main database during development. This will start the emulator on port 8083 and the app server on port 8000.

*   **Start with Specific Environment (e.g., `dev`):**
    ```bash
    APP_ENV=dev ./scripts/start-local-server.sh
    ```
    This will load configuration from `.env.dev`.

### `start-local-docker.sh`

Builds the application Docker image and runs it locally. This simulates the Cloud Run environment. This script is intended for testing the Dockerfile setup and debugging potential deployment issues before pushing to Cloud Run. It provides a local environment that closely mimics Cloud Run.

**Usage:**

```bash
./scripts/start-local-docker.sh
```
*   Builds image `izumi-backend:dev`.
*   By default, the Docker image is built for the `dev` environment (`APP_ENV=dev`) as configured in the Dockerfile.
*   Runs on port 8080.
*   Mounts your local gcloud credentials (`~/.config/gcloud`) into the container so the app can authenticate with GCP services.


## Infrastructure & Deployment

### `setup_gcp_project.py`

This script is designed for users of this project to deploy their own instance onto a Google Cloud project. It's a comprehensive tool that handles all necessary infrastructure setup, leveraging Terraform under the hood to configure resources based on the provided project details. For detailed instructions, refer to `deployment/README.md`.

**Usage:**

*   **Local Development Setup:**
    ```bash
    uv run scripts/setup_gcp_project.py
    ```
    Creates a `.env.local` file for local testing.

*   **Cloud Environment Setup (e.g., dev, prod):**
    ```bash
    uv run scripts/setup_gcp_project.py --app_env dev
    ```
    Provisions resources in GCP and creates `.env.dev`.

*   **Destroy Environment:**
    ```bash
    uv run scripts/setup_gcp_project.py --app_env dev --destroy
    ```
    Destroys all resources associated with the specified environment.

### `deploy-to-cloud-run.sh`

Deploys the application to Google Cloud Run using Cloud Build. It relies on the configuration generated by `setup_gcp_project.py`. This script is used to deploy the latest codebase from the repository to a Cloud Run instance that has been previously set up by `setup_gcp_project.py`.

**Usage:**

```bash
./scripts/deploy-to-cloud-run.sh --app_env <ENVIRONMENT>
```

**Example:**
```bash
./scripts/deploy-to-cloud-run.sh --app_env dev
```
*   Requires `.env.<ENVIRONMENT>` to exist.
*   Submits a build to Cloud Build using `deployment/cloudbuild.yaml`.

## Utilities


### `archive-repo-as-zip.sh`

Creates a zip archive of the current repository state. This script creates a self-contained ZIP file of the entire repository. This ZIP file can then be uploaded to environments like Cloud Shell for easy deployment of the project.

**Usage:**

```bash
./scripts/archive-repo-as-zip.sh
```
*   Output: `mediagent_demos.zip` in the project root.

### `install_gerrit_hooks.sh`

Installs the Gerrit commit-msg hook in the main repository. The hook injects a Change-Id into commit messages, which is essential for how Gerrit Code Review tracks and manages changes.

**Usage:**

```bash
./scripts/install_gerrit_hooks.sh
```
